How Portico works.

A line-by-line tour of an Electron + Claude Agent SDK desktop app. Read with the repo open — every section links to the file that does the work.

Three processes, one app

Electron is two tools in one binary: Node.js (full OS access — read files, spawn shells) and Chromium (a sandboxed browser engine that renders your UI). When Portico launches it spawns several OS processes that play different roles. Knowing which code runs where is the single most important Electron concept.

Process #3 — Chromium

Renderer

The React app you see. Runs in a sandbox like a webpage. Cannot touch the filesystem, the network outside HTTP, or the API key.

Process #2 — bridge

Preload

The trust boundary. Defines exactly which functions the renderer can call. Anything not exposed here is invisible to the UI.

Process #1 — Node

Main

The privileged process. Owns the API key, talks to the Claude SDK, reads/writes files, spawns the agent's tools.

Why split it like this?

Security. The renderer renders untrusted-looking content — your prompt, the model's markdown response, third-party React libraries. If any of that has a script-injection bug, you do not want it to also have permission to fs.readFileSync('/etc/passwd') or exfiltrate process.env.ANTHROPIC_API_KEY. Chromium's sandbox makes that impossible by default. Anything the renderer needs from the OS goes through the IPC bridge below — and we control exactly what's exposed.

What happens when you launch the app

Trace the boot, top to bottom.

01
Electron reads package.json and starts the main entry.
"main": "./out/main/index.js"
In dev mode (npm run dev) electron-vite rebuilds and re-launches on every file change. In production this points at the bundled main process.
02
Main process loads .env, then user settings.
src/main/index.ts:12 + appSettings.applyToEnv()
dotenv first puts .env values into process.env. Then on app.whenReady we call appSettings.applyToEnv() which overwrites with the user's saved gateway URL + API key (from the encrypted settings.json). App settings always win over .env — that's the contract.
03
Main registers ~25 IPC endpoints.
ipcMain.handle('agent:query', ...)
These are the only functions the renderer can call. Each is a typed promise. Examples: agent:query, conversations:list, dialog:pickFolder, permission:respond.
04
Main creates a BrowserWindow.
webPreferences: { contextIsolation: true, nodeIntegration: false, ... }
A window is essentially a Chromium tab. The crucial config is the preload path — that script runs before the page loads, in a privileged context that has both Node and the upcoming window object.
05
The preload script runs and builds the bridge.
contextBridge.exposeInMainWorld('api', api)
After this line, the renderer can do window.api.query(...). It cannot import electron, cannot touch fs, cannot see ipcRenderer directly. Only what we put on the api object is reachable.
06
The HTML loads and React mounts.
src/renderer/src/main.tsx → <App />
From here on, everything is a React app. It looks like a webpage, behaves like a webpage. The only unusual thing is the global window.api object — that's the door to everything else.

The IPC dance

How a renderer-side function call ends up running Node code in main, and how the response gets back. Every Portico feature uses this same three-step pattern.

1 · Renderer calls api
// src/renderer/src/App.tsx
await window.api.query(prompt, runId, conversationId)

The renderer doesn't know — and can't know — what's behind that function. It's a Promise that returns whenever the response comes back.

2 · Preload forwards via ipcRenderer.invoke
// src/preload/index.ts
query: (prompt, runId, conversationId) =>
  ipcRenderer.invoke('agent:query', prompt, runId, conversationId)

invoke sends a message over Electron's IPC channel and returns a Promise that resolves with main's return value.

3 · Main handles the call
// src/main/index.ts
ipcMain.handle('agent:query', async (event, prompt, runId, conversationId) => {
  // validate inputs, call Claude SDK, stream events back…
})

Main does whatever it needs — read files, call the SDK, spawn a process. The Promise it returns becomes the renderer's resolved value.

Streaming the other way
// main pushes stream events back
sender.send('agent:message', payload)

// renderer subscribes via the bridge
window.api.onMessage((runId, msg) => { ... })

For things like streaming tokens, main pushes events via send; the renderer registered listeners during init. Same channel, opposite direction.

The string channel names ('agent:query') are the contract. If main and preload disagree on a name, the call silently fails. That's why the convention is namespace:verb and why the preload re-exports types from src/shared/types.ts — so the renderer's call signature and main's handler can never drift.

What's actually happening underneath

  1. You hit Enter. React adds a user bubble to state and calls window.api.query(prompt, runId, conversationId).
  2. Preload turns that into ipcRenderer.invoke('agent:query', ...). The call crosses the process boundary.
  3. Main looks up the conversation's saved cwd and sessionId, builds an AbortController, and calls the Claude Agent SDK's query({ prompt, options }).
  4. The SDK calls the gateway (Portkey or Anthropic direct). Because includePartialMessages: true, the SDK starts emitting stream_event messages immediately as tokens arrive.
  5. Main forwards each event to the renderer via sender.send('agent:message', ...). The renderer's onMessage callback applies them to the current assistant bubble's blocks array — text grows, thinking fills in, tool_use pills appear.
  6. If the agent invokes a tool like Bash, and you're in Ask mode, main pauses the SDK via the canUseTool callback — sends a permission:request event, awaits a permission:respond from the renderer.
  7. Tool runs, result comes back as a user-role message with tool_result content. The renderer renders a collapsible card.
  8. Agent does another assistant turn with the tool output in context — gets a fresh message_start with a new message.id, which means a new bubble.
  9. Eventually the SDK finishes. Main sends agent:done. Renderer's debounced effect saves the bubbles to disk 400ms later.

What we pass to the Claude Agent SDK

Every query() call is built from these options. Here's what each one does.

model claude-sonnet-4-6 by default. Override via PORTICO_MODEL. Sonnet is the right tier for chat — Opus when you need heavier reasoning, Haiku when you want fast and cheap.
systemPrompt By default, extends the claude_code preset (the SDK's full agent prompt) with a Portico identity line. PORTICO_PLAIN_SYSTEM_PROMPT=1 drops the preset for a faster, less agentic model.
permissionMode 'ask' in first-run defaults — every tool call pauses for your approval.
'bypassPermissions' when you set Settings → Permissions to "Auto-approve everything" — every tool runs immediately.
canUseTool Set in Ask mode. A callback that fires before each tool call. Returns { behavior: 'allow' } or { behavior: 'deny', message }. We use it to surface the inline approval card and (optionally) run a Haiku screening pass first.
includePartialMessages Always true. Without it, you'd wait for the entire assistant turn before seeing any text — instead the SDK emits stream_event messages with content_block_delta events that we apply token-by-token.
cwd The conversation's working directory (chosen via the right sidebar). The agent's Read / Write / Bash tools all run relative to this. Re-validated at query time — if the folder was deleted between sessions, we drop the cwd silently.
resume The SDK's session_id from the previous turn. Without this, multi-turn memory breaks — every message would be a fresh context. Captured from the first system.init message and stored on the conversation.
abortController An AbortController per run. Hitting the Stop button calls controller.abort() in main, which exits the SDK's stream cleanly.

Working directory + filesystem access

The agent has the full toolset: Read, Write, Edit, Bash, Glob, Grep, etc. None of these are special — the SDK executes them in the main process using normal Node APIs (fs.readFile, child_process.spawn).

How "open another directory" actually works

  1. You click "Choose folder…" in the right sidebar.
  2. Renderer calls window.api.dialog.pickFolder().
  3. Main opens Electron's native dialog.showOpenDialog with properties: ['openDirectory']. The user picks a folder.
  4. Main returns the path. Renderer calls conversations.setCwd(id, path).
  5. Main validates: must exist, must be a directory, must be inside $HOME (the isCwdSafe guard). Persists to conversations.json.
  6. The next query() call passes that path as options.cwd. The SDK runs every tool relative to it.

Why the validation matters

Without isCwdSafe, a compromised renderer could call setCwd('/etc') and the next Bash tool call would happily run there. The same guard runs at write time (conversations:setCwd handler) and at read time (in agent:query) so a tampered conversations.json file can't bypass it either. Belt + suspenders.

The permission dance (Ask mode)

In Ask mode the agent can't touch a tool without your nod. Optionally, a Haiku call screens the request first.

Inside main

Per query, we pass a canUseTool async callback. Each tool the SDK wants to call awaits this: 

async canUseTool(toolName, input, opts) {
  const requestId = `${runId}:${uuid()}`
  if (settings.autoScreen) {
    send('permission:screening', { requestId, toolName })
    screening = await screenTool(toolName, input, cwd)
  }
  send('permission:request', { requestId, toolName, input, screening })
  const { allow } = await new Promise(...)
  return allow ? { behavior: 'allow' } : { behavior: 'deny', message: 'denied' }
}
Inside renderer

The renderer treats permission:screening and permission:request like any other stream event. It adds a special "permission" bubble to the chat.

Security at a glance

What's protecting you from a bad prompt or a compromised renderer.

Renderer sandbox Chromium runs in a sandbox. contextIsolation: true, nodeIntegration: false — the page can't require('fs') even if it wanted to.
API key boundary The key lives in process.env in main. The renderer never sees it as plaintext after first save — it only learns { gatewayKeySet: boolean }. Stored on disk encrypted via your OS keychain (safeStorage).
URL allowlist Markdown links go through shell.openExternal only if their scheme is https, http, or mailto. file://, vscode://, slack://, javascript: — all blocked.
Path validation isCwdSafe blocks the agent from running outside your home folder. isPathRevealable blocks Finder side-effects from probing system paths. Validated server-side.
Tool permission First-run defaults to "Ask" mode — every tool waits for your approval. Optional Haiku screening (~300ms) tags each call Safe / Caution / Dangerous before showing the prompt.
Settings whitelist The appSettings.set handler only writes permissionMode, autoScreen, gatewayBaseUrl. A renderer compromise can't poison arbitrary keys into settings.json.
Atomic writes + corrupt-file backup Every settings/conversation write goes tmp file → rename. If conversations.json ever fails to parse, we move it aside as .corrupt-<ts> instead of overwriting with {}.
90 unit tests The guards, the persistence layer, the API key handling, the screening parser — all covered in tests/. Run with npm test.

What's still risky and why

The agent runs with the full toolset. Even with Ask mode, if you click Approve on a Bash rm -rf ~ request, that runs. The screening verdict is advisory — the model is smart but not infallible. Treat Approve like running the command yourself.

For multi-tenant or external distribution: don't ship with bypassPermissions as the default. The first-run default of Ask + Auto-screen is the safe shape; Auto mode should be opt-in.
