Where everything lives.

A tour of every source file in Portico — what it does in one line, when you'd touch it, and a "where do I change X?" lookup at the bottom. If you're presenting tomorrow, this is the page that lets you confidently answer "where would I add that?"

Companion to architecture.html (how it works) and flows.html (what happens when). This doc is about geography — where files live, what they own, what to touch when you want to change something.

Why three processes (the trust boundary)

Portico is an Electron app. Electron splits your code across three processes for a security reason — not just because it's tidy. The split is the entire reason the API key never touches the browser.

Main
Privileged
Full Node.js access. Owns the API key, the SDK, the filesystem, native dialogs.
⇄
Preload
The bridge
Tiny script. Whitelists which IPC channels the renderer can call. Defines window.api.
⇄
Renderer
Sandboxed
Browser-style. Renders attacker-controllable Markdown. Cannot touch Node, the SDK, or the key.

Why this matters: the agent generates Markdown that you render in the chat bubbles. If a malicious tool result included a script tag (or some clever injection), sandboxing the renderer means it can't reach your API key, can't read arbitrary files, and can't run shell commands. Every capability the renderer wants (read a file, run a query, pick a folder) goes through a typed IPC channel the preload explicitly exposes.

Mental rule of thumb: "if it's secret or powerful, it's in main; if it's drawn on screen, it's in renderer; the preload is just the doorway."

Main process · src/main/

Node.js · privileged

The brain. Holds the API key, talks to the Claude Agent SDK, owns persistence, registers every IPC handler. ~9 files, none crazy big except index.ts.

main/index.ts
The orchestrator. Creates the BrowserWindow, registers all IPC handlers, runs the agent:query loop with the SDK, owns the canUseTool callback, manages cancellation, holds the active runs map. adding/changing IPC handlers, the agent loop, the permission flow, model resolution.
680 lines
main/conversations.ts
The conversation store. JSON file at <userData>/portico/conversations.json. All metadata setters (sessionId, cwd, trustProject, model, sessionAllowedPatterns) go through the updateConversation() helper. Sidebar sort field lastMessageAt is ONLY bumped by save() — not by any setter. adding a new per-conversation field, changing how sessions persist, sidebar order rules.
245 lines
main/appSettings.ts
App-wide settings (gateway URL, API key, permissionMode, autoScreen, defaultModel). Persists at <userData>/portico/settings.json. API key encrypted via Electron safeStorage when keychain available, plaintext fallback (warned in UI). Has a WRITABLE whitelist — any new field you want the renderer to set must be added here. adding a new global setting, changing key storage, gateway env application.
152 lines
main/userSettings.ts
Manages files under ~/.claude/: CLAUDE.md (memory), soul.md (response style), skills/. Owns the persona wizard's seeding logic, reads back the user's name for the greeting. readSoul() drives the system-prompt append. changing how persona wizard works, editing the CLAUDE.md/soul.md schema, skill CRUD.
245 lines
main/screenTool.ts
The Haiku auto-screen. Given a tool call, classifies SAFE / CAUTION / DANGEROUS in ~300ms with a structured Anthropic API call. Caches its client; invalidates on gateway settings change. tuning the screening prompt, changing the screening model, adding new verdicts.
108 lines
main/permissionPatterns.ts
Per-session allowlist matcher. Pattern syntax: Bash(python *), Read(/foo/**), bare Edit. Glob → regex with ** crossing slashes, * not. suggestPattern() generates the "Allow for session" recommendations. extending the pattern syntax, adding new matchable tool fields.
131 lines
main/guards.ts
Path + URL safety checks. isCwdSafe() (no symlinks out of home), isPathRevealable() (only paths under home or userData), isExternalSchemeAllowed() (only https/http/mailto for shell.openExternal). Used by every IPC handler that touches the filesystem or shell. adding a new path-based IPC, allowing new URL schemes.
50 lines
main/logger.ts
In-memory ring buffer (500 entries) + daily-rotated log files at <userData>/portico/logs/portico-<YYYY-MM-DD>.log. Broadcasts logs:appended events to all BrowserWindows so the Settings → Advanced tab shows live logs. changing log retention, adding structured fields, switching log destinations.
123 lines
main/util.ts
Two helpers: porticoDir() (returns the userData/portico dir, creates if missing) and atomicWrite(path, content) (write to .tmp, then rename — never leaves a half-written file if the process crashes mid-write). adding new file persistence; reach for atomicWrite, don't use writeFileSync directly.
23 lines

Preload · src/preload/

The bouncer

The smallest of the three processes. One file. Its only job: define exactly which functions the renderer can call on window.api. Nothing else gets through.

preload/index.ts
Wraps every ipcRenderer.invoke(channel, ...) in a typed function and exposes them via contextBridge.exposeInMainWorld('api', { ... }). Re-exports types from shared/types.ts — type-only re-exports! Runtime values must come from shared/types directly, otherwise vite tries to bundle the preload (which depends on Electron) into the renderer (which doesn't have Electron) and the page goes blank. adding a new IPC channel: handler in main/index.ts → typed wrapper here → use as window.api.foo.bar() in renderer.
242 lines

How to add a new IPC channel

  1. Type — if anything new crosses the boundary, define it in src/shared/types.ts.
  2. Handler — register in src/main/index.ts: ipcMain.handle('namespace:verb', async (_e, ...) => { ... }).
  3. Wrapper — add a typed function under api in src/preload/index.ts.
  4. Use — call from renderer as await window.api.namespace.verb(...).

Channel names use namespace:verb (e.g. conversations:list, agent:cancel).

Shared · src/shared/

Cross-process types

The only file imported across the trust boundary. Pure TypeScript — no Node, no Electron, no React. If a type travels between main, preload, and renderer, it lives here.

shared/types.ts
All shared types: Block, Bubble, Conversation, AppSettings, PermissionRequest, Screening, UserQuestion, UserQuestionAnswer, etc. Plus runtime constants like KNOWN_MODELS, DEFAULT_CONTEXT_WINDOW, and helper functions (lookupModel, contextWindowFor) that all three processes can use. adding a new field that crosses the boundary, adding a new model to the picker, anything that needs to be the same shape on both sides.
243 lines

Gotcha: renderer code can import types from ../../preload (they're erased at compile time), but runtime values like KNOWN_MODELS must come from ../../shared/types directly. Importing a runtime value from preload drags the preload's contextBridge calls into the renderer bundle and crashes the page on load. We learned this the hard way.

Tests · tests/

Vitest. 130 tests across 7 files. All main-process modules; renderer is tested manually for now.

tests/setup.ts
Mocks electron (safeStorage, app.getPath, dialog). Per-test scratch userData + HOME dirs so writes don't leak between tests.
tests/conversations.test.ts
Conversation store CRUD + migration. Verifies upsert behavior, sort field invariants, atomic write recovery.
tests/appSettings.test.ts
Settings persistence, encrypted vs. plaintext key storage, applyToEnv push/pull, WRITABLE whitelist enforcement.
tests/userSettings.test.ts
CLAUDE.md / soul.md / skills CRUD. Persona seeding writes the right "About me" section.
tests/screenTool.test.ts
Haiku screening output parsing, error fallback to CAUTION, client cache invalidation on settings change.
tests/permissionPatterns.test.ts
Pattern parser, glob → regex, matches() per-tool target extraction, suggestPattern round-trip.
tests/guards.test.ts
URL scheme allowlist, cwd-safe path checks, reveal-path home/userData boundary.
tests/util.test.ts
atomicWrite + porticoDir.

Pattern: every test does dynamic await import(...) inside its it() block. Module-scope state in the SUT (caches, env reads) gets a fresh slate per test instead of leaking across them.

"Where do I change X?"

The reference table. If someone asks "how would you add Y?" in the demo tomorrow, find the row that matches their question. Type to filter.

Add a new model option (e.g. Sonnet 5)
Add a ModelOption entry to KNOWN_MODELS in src/shared/types.ts. Picker UI, default dropdown, gateway badge, context meter all read from there — they update automatically.
Add a new global setting (something user toggles in Settings)
1. Add the field to AppSettings in src/shared/types.ts.
2. Add the key to the WRITABLE set in src/main/appSettings.ts.
3. Add the UI in src/renderer/src/components/Settings.tsx — pick the right tab or add a new one.
Add a new per-conversation field (something stored per chat)
1. Add the field to Conversation in src/shared/types.ts.
2. Add a setField() helper in src/main/conversations.ts using the updateConversation() helper — one line.
3. New IPC handler in src/main/index.ts: conversations:setField.
4. Add to conversations in preload.
5. Use from renderer as window.api.conversations.setField(id, value).
Add a new tool (via SDK MCP)
Define an SDK MCP server in TS (small in-process MCP) and pass it via mcpServers in the query() options inside src/main/index.ts's agent:query handler. The tool then flows through the existing canUseTool permission UI for free.
Customize the AskUserQuestion modal
UI: src/renderer/src/components/UserQuestionModal.tsx. Wiring (parsing the SDK input, formatting the answer back to the agent): parseAskUserQuestionInput() and formatUserAnswers() in src/main/index.ts.
Change how tool calls render (compact log style, diff view, etc.)
ToolUseBlock + summarizeToolCall() + ToolDetail + DiffView all in src/renderer/src/components/BubbleView.tsx.
Change the system prompt or its layers
systemPromptFor(), buildAppend(), PORTICO_APPEND, PROFILE_FILES_CODA in src/main/index.ts. soul.md content lives in ~/.claude/soul.md and is loaded by userSettings.readSoul().
Add a new permission-mode option (e.g. plan mode)
1. Add to PermissionMode union in src/shared/types.ts.
2. Add the corresponding branch in agent:query's options block in src/main/index.ts (e.g. set disallowedTools for plan mode).
3. Surface in Settings → Permissions tab in Settings.tsx.
Change the auto-screen prompt or model
src/main/screenTool.ts. Prompt is at the top of screenTool(); model is the SCREEN_MODEL constant.
Allow a new pattern syntax for session permissions
src/main/permissionPatterns.ts. Update PATTERN_RE, parse(), globToRegex() as needed. Add tests to tests/permissionPatterns.test.ts.
Change how messages stream into bubbles
applyStreamEvent() in src/renderer/src/App.tsx. The canonical assistant-message handler (which backfills tool inputs at end-of-turn) is right below it.
Change the working folder picker / trust flag
UI: src/renderer/src/components/RightSidebar.tsx. Backend: setCwd / setTrustProject in src/main/conversations.ts + the corresponding IPC handlers in src/main/index.ts (conversations:setCwd, conversations:setTrustProject).
Change the message composer (input, send button, mascot)
src/renderer/src/components/Composer.tsx. Pet positioning is the absolute wrapper at the top of the JSX. Send/Stop buttons are inline at the right.
Change the mascot (color, shape, mood)
src/renderer/src/components/Pet.tsx. Color tokens at the top of the file. Each mood (idle/thinking/happy) has its own SVG group.
Add a new sidebar item or change recents sort
UI: src/renderer/src/components/Sidebar.tsx. Sort field is lastMessageAt, computed in src/main/conversations.ts:list().
Add a new IPC channel
1. Define the type (if anything new crosses the boundary) in src/shared/types.ts.
2. Register the handler in src/main/index.ts: ipcMain.handle('namespace:verb', ...).
3. Expose a typed wrapper under api in src/preload/index.ts.
4. Use from renderer as await window.api.namespace.verb(...).
Add a new env var
Read in src/main/index.ts (only main has process.env access). Document in CLAUDE.md env table AND .env.example. Surface to renderer via gateway:info or a new IPC handler — never expose secrets directly.
Add a new content block type for bubbles (e.g. image, table)
1. Extend the Block union in src/shared/types.ts.
2. Render branch in src/renderer/src/components/BubbleView.tsx.
3. If it streams, add a handler in applyStreamEvent (App.tsx).
4. If it appears in old conversations, add a fallback in src/main/conversations.ts:migrate.
Add a new Settings tab
1. Add to Tab union at the top of src/renderer/src/components/Settings.tsx.
2. Add a <TabBtn> to the nav and a render branch to the body.
3. Write a function NewTab(): JSX.Element sub-component in the same file.
4. If it persists state, hit the "Add a new global setting" recipe above.
Change the persona wizard or its templates
UI: src/renderer/src/components/PersonaWizard.tsx. Templates and seeding logic: SOUL_TEMPLATES + seedProfile() in src/main/userSettings.ts.
Change tour steps or copy
STEPS array at the top of src/renderer/src/components/Tour.tsx. Anchor refs are forwarded from App.tsx.
Change logging (retention, structure, destination)
src/main/logger.ts. Live-log viewer in Settings → Advanced is in Settings.tsx's AdvancedTab.
Allow a new external URL scheme
isExternalSchemeAllowed() in src/main/guards.ts. Add to the allowlist and add a case to tests/guards.test.ts.
Bundle a new built-in skill
Drop a SKILL.md with frontmatter into resources/bundled/skills/<name>/. To load it, the conversation needs cwd set to a workspace dir that includes it OR pass via settingSources / skills in agent:query.
Change branding / app name (window title, etc.)
Window title in src/main/index.ts (BrowserWindow options). User-visible strings are in: App.tsx (header "Powered by Claude", greeting, empty states), Sidebar.tsx (wordmark), electron-builder.yml (productName), README. Internal slugs (CCD, com.portico.app, package name) stay per the branding rule.

Quick recap: the harness lens

Portico is a harness — wraps the Claude Agent SDK with controlled mediation between human and agent. Four mediation surfaces; every feature lives at one. Useful when deciding "where does this go?"

What the agent sees
System prompt, settingSources, cwd, prepended user-prompt notes. main/index.ts (systemPromptFor, buildAppend), main/userSettings.ts (soul.md, CLAUDE.md).
What the agent can do
Tool allowlist, canUseTool callback, mcpServers, permissionMode. main/index.ts (canUseTool body), main/permissionPatterns.ts, main/screenTool.ts.
When it acts
Who triggers a turn (today: just the user via Send button). main/index.ts (agent:query handler is the only trigger today; future heartbeat lands here).
How the human stays in the loop
Bubbles, modals, badges, observable state, header info, mascot. renderer/components/BubbleView.tsx, PermissionPrompt, UserQuestionModal, Settings, Pet, Composer.
