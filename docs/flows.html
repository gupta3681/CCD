Flows & hooks

Portico is a harness — it wraps the Claude Agent SDK with controlled mediation between the human and the agent. This doc walks the four mediation surfaces: what the agent sees (system prompt), what it can do (permissions), how the answer comes back, and where future hooks would slot in.

Companion to architecture.html. This doc is about movement — the order of events through the harness. The architecture doc is about placement — what lives where.

1 — How a permission decision is made

The agent is about to call a tool (Read, Bash, Edit, …). What happens between "agent says I want to run X" and "X actually runs" or "X is blocked"?

Main process Renderer Agent SDK Decision point User input
Step 0 / 8
1
SDK fires canUseTool(toolName, input, opts)
The Agent SDK is about to execute a tool, calls our callback for permission first. Includes opts.signal AbortSignal.
node_modules/@anthropic-ai/claude-agent-sdk → main/index.ts agent:query handler
2
Is this AskUserQuestion?
Special case: bypass the entire permission flow, surface a structured modal instead.

3
Does any session-allowed pattern match?
Check conversation.sessionAllowedPatterns[] against this call. Bash(python *) matches Bash with command starting python .

Is auto-screen on?
If yes, send permission:screening to renderer (placeholder bubble), then call Haiku to classify SAFE / CAUTION / DANGEROUS (~300ms).

5

Bash(python *) for Bash, Read(/foo/*) for Read, bare tool name for everything else.

Carries: requestId, toolName, input, screening verdict, suggested pattern. Renderer renders the permission card with screening bar + 3 buttons (Deny / Approve once / Allow for session ▾).

The dual short-circuits at steps 2 & 3 are deliberate — both skip the entire permission UI. AskUserQuestion gets its own modal (it's not really a permission, it's a structured ask). Allowlist matches mean "user already decided this kind of thing is fine" — no point asking again the same session.

2 — How the system prompt is assembled

Three ways to set it: full env override, a Portkey-friendly plain-text mode, or the default "claude_code preset + Portico append." The append itself is layered. Here's the actual composition order:

PORTICO_SYSTEM_PROMPT
If set, this string is the entire system prompt. Hard override — none of the layers below run.
main/index.ts systemPromptFor()
claude_code preset
Default. The full Anthropic-bundled agent preset that gives the model its tool guidance, formatting expectations, etc. Skipped if PORTICO_PLAIN_SYSTEM_PROMPT=1.
SDK option systemPrompt: { type: 'preset', preset: 'claude_code', append }
PORTICO_APPEND
Identity line: "You are Portico, a desktop assistant…". Sets formatting expectations (markdown, concise, immediate over clarifying).
main/index.ts PORTICO_APPEND constant
soul.md (if exists)
The user's response-style rules. Wrapped in "## How to respond". Only included if ~/.claude/soul.md exists with non-empty content.
main/userSettings.ts readSoul() · main/index.ts buildAppend()
PROFILE_FILES_CODA
Final paragraph telling the agent it can edit ~/.claude/CLAUDE.md and soul.md to keep its context current.
main/index.ts PROFILE_FILES_CODA constant

Per-turn additions to the user prompt

The system prompt above is fixed for the conversation. The user prompt for each turn is also massaged:

1
User types and hits Send
Raw prompt string from the textarea.
renderer App.send()
2
Was the previous turn interrupted?
If conversation.lastInterrupted === true, prepend a short note so the model knows it was cut off. Then clear the flag.
main/index.ts agent:query · conversations.getInterrupted()/setInterrupted()
3
Resolve cwd, trustProject, settingSources
trustProject true → settingSources is ['user', 'project'], otherwise ['user']. Project's .claude/ is opt-in for prompt-injection safety.
main/index.ts agent:query handler
4
Resolve model — precedence chain
conversation.model > appSettings.defaultModel > PORTICO_MODEL env > claude-sonnet-4-6.
main/index.ts modelFor(conversationId)
5
SDK query() with the assembled options
Pass model, systemPrompt, includePartialMessages, abortController, settingSources, toolConfig, canUseTool, optional resume. Returns an async iterator of messages.
@anthropic-ai/claude-agent-sdk · query()

3 — How the answer comes back

Streaming. Lots of message types. The renderer assembles them into bubbles indexed by SDK message.id so each agent turn becomes one visual bubble even though tool-use loops produce many turns.

SDK iterator yields

system: init
stream_event: message_start
stream_event: content_block_start (text)
stream_event: content_block_delta × N
… more deltas …
stream_event: content_block_stop
stream_event: content_block_start (tool_use)
input_json_delta × N (we ignore)
stream_event: content_block_stop
stream_event: message_stop
assistant: { message.content[], usage }
user: { tool_result }
… new turn for tool_use → assistant ↻ …
result

Main: agent:query loop

activeRuns.set(runId, {...})
setSessionId on first init
send('agent:message', message)
… for each yielded message …
send('agent:done', null)
activeRuns.delete(runId)

Preload bridge

contextBridge → window.api.onMessage
forwards each event to renderer

Renderer: applyStreamEvent

message_start → currentMessageIdRef.set
content_block_start → blocks[idx] = {type, text:''}
content_block_delta (text) → append to text
content_block_delta (thinking) → append to thinking
tool_result → push 'tool' bubble
assistant message → backfill canonical tool inputs
assistant.usage → setContextTokens
agent:done → setBusy(false)

Why we backfill tool inputs from the canonical assistant message

During streaming, tool inputs arrive as input_json_delta events whose partial_json strings concatenate to the full JSON. We intentionally ignore those deltas in applyStreamEvent — parsing half-formed JSON every keystroke is fiddly and error-prone. Instead, when the SDK emits the canonical assistant message at end-of-turn (which has the fully-parsed tool_use.input), we walk that content[] array and overwrite the blocks in our bubble. Net effect: a brief moment where the tool row shows Edited file +0 -0, then it snaps to Edited barcelona.txt +12 -3 when the turn closes. Tradeoff accepted.

See renderer/src/App.tsx applyStreamEvent + the assistant-message handler.

4 — Where to add hooks

The current code is direct — no hook system, just inline logic in the agent:query handler. There are five natural seam points where future code (Buddy auto-observe, heartbeat ticks, post-turn telemetry, custom screening) would slot in.

When
What you'd do here
preTurn Before query() fires
Modify effectivePrompt (e.g. inject reminders, prepend the interrupt note), modify systemPrompt append, set extra mcpServers, set extra settingSources. Today: only the interrupt-prepend is here, inline.
preToolUse Inside canUseTool, before sending to renderer
Short-circuit allow/deny without a prompt (e.g. policy "never run rm -rf without explicit confirm even if Bash is allowlisted"). Today: only the session allowlist + AskUserQuestion intercept are here.
postToolUse After SDK yields user: {tool_result}
Notification on tool errors, post-tool linter ("you just edited X — also update Y"), per-tool cost accounting, audit trail. Today: nothing here — tool results just get rendered.
postTurn After SDK iterator drains, before agent:done
Buddy buddy_observe call, telemetry, auto-archive idle conversations. Today: nothing — main just sends agent:done and unwinds.
onMessage Inside the streaming for-loop, per yielded message
Custom message-type handlers (the SDK has SDKPermissionDeniedMessage, SDKCompactBoundaryMessage, etc. that we currently pass through to the renderer verbatim). Hook here for first-class handling.

The shape these hooks should take when extracted

// proposed: src/main/harness/hooks.ts
interface TurnContext {
  runId: string
  conversationId: string
  prompt: string
  systemAppend: string
  cwd?: string
  trustProject: boolean
  model: string
  mcpServers: Record<string, unknown>
}

type PreTurnHook     = (ctx: TurnContext) => Promise<TurnContext>
type PreToolHook     = (ctx: TurnContext, name: string, input: unknown)
                          => Promise<{ allow: boolean, reason?: string } | null>
type PostToolHook    = (ctx: TurnContext, name: string, input: unknown, result: string)
                          => Promise<void>
type PostTurnHook    = (ctx: TurnContext, summary: { sdkEvents: number, error?: Error })
                          => Promise<void>

export const hooks = {
  preTurn:     [] as PreTurnHook[],
  preToolUse:  [] as PreToolHook[],
  postToolUse: [] as PostToolHook[],
  postTurn:    [] as PostTurnHook[]
}

Backed by the compartmentalization plan below — extract harness/runner.ts, harness/promptAssembler.ts, and harness/hooks.ts, then everything in index.ts becomes thin IPC dispatch.

Bug review — May 10, 2026

Walked the recent changes (model picker, AskUserQuestion modal, sessionPermissions, tool result rendering). Two real bugs found and fixed in this pass; three more flagged as pre-existing and noted below.

Fixed in this session

AbortSignal listener leak
canUseTool added an 'abort' listener but never removed it on natural resolve. Long turns with many tool calls accumulated listeners. Fixed via new awaitPermissionOrAbort() helper that single-resolves and removes the listener.
main/index.ts (was duplicated in two places)
Silent loss of session_id on init
If the SDK's system:init ever lacked session_id, we silently skipped setSessionId(). The conversation would then run as single-turn-only with no warning. Fixed: log.warn if init has no session_id.
main/index.ts agent:query streaming loop

Pre-existing — flagged for backlog

activeRuns leak on iterator stall
If the SDK's async iterator hangs and never throws, activeRuns.delete(runId) in finally never runs. Long-lived Portico instances accumulate orphaned entries. Mitigation: wrap the for-await with a watchdog timeout, or surface stalls via a heartbeat-like check.
main/index.ts agent:query for-await loop
UserQuestionModal selections race
Modal initializes selections from questions length. If questions prop ever changes mid-mount (currently it can't, we set askUserQuestion to null between requests), selections[i] would index out of bounds. Mitigation: key={requestId} on the modal in App.tsx.
renderer/src/components/UserQuestionModal.tsx
gatewayInfo IPC fires twice on mount
Both the main IPC-subscriptions effect AND the conversationId-deps effect fetch gateway info. Wasteful but not broken. Mitigation: drop the call from the main effect, let the deps effect own it.
renderer/src/App.tsx

Verified against the actual source (not just LLM hunches). Six items the review agent flagged were not real bugs — defensive-coding misreads. Net: 2 fixed, 3 noted, no surprises.

Compartmentalization plan

The codebase is healthy at v1, but two files are doing too much: main/index.ts (~700 lines, 25+ IPC handlers + the harness loop) and renderer/src/App.tsx (~750 lines, all the orchestration). Here's the proposed split when we have a clean half-day. Pure refactor — no behavior change. The new harness/ directory makes the framing explicit: the harness IS what Portico ships.

main/harness/runner.ts
The actual SDK query() loop. ~50 lines. Takes a TurnContext, returns nothing (events go via send callback). Replaces the body of agent:query.
extract from main/index.ts
main/harness/promptAssembler.ts
Pure functions: buildSystemPrompt(), buildUserPrompt(), resolveModel(), resolveSettingSources(). No side effects, no IPC. Trivially testable.
extract from main/index.ts
main/harness/permissionGate.ts
The canUseTool body — currently 100 lines of nested logic. Splits into: handleAskUserQuestion(), checkAllowlist(), screenAndPrompt().
extract from main/index.ts
main/harness/hooks.ts
Hook registry (preTurn / preToolUse / postToolUse / postTurn). Today the inline logic is the "first registered hook" for each point. Future features register additional hooks.
new file
main/ipc/conversations.ts
All conversations:* IPC handlers. Just delegates to conversations.ts. ~30 lines, removes IPC noise from index.ts.
extract from main/index.ts
main/ipc/settings.ts
All settings:*, appSettings:*, logs:*, buddy:* IPC handlers.
extract from main/index.ts
renderer/src/hooks/useStreaming.ts
Custom React hook owning currentMessageIdRef, applyStreamEvent, the assistant-message backfill, and the bubble-state mutations. App.tsx becomes orchestration only.
extract from renderer/src/App.tsx
renderer/src/hooks/useConversation.ts
Owns conversationId, bubbles, cwd, trustProject, model, sessionPatterns. Provides newSession, selectSession, setModel, etc. as a single hook.
extract from renderer/src/App.tsx
renderer/src/components/Composer.tsx
The textarea + Pet + send button as one component. App.tsx renders <Composer onSend={send} busy={busy} onStop={stop} />.
extract from renderer/src/App.tsx
